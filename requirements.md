# OmniAuto 需求规格说明书 (SRS)

> **版本**: v1.0  
> **项目定位**: 面向 Vibe Coding 的通用 AI 自动化中间件，通过自然语言描述即可生成浏览器/桌面软件自动化工作流，作为**影刀 RPA 的开源替代方案**。  
> **核心哲学**: 人类用自然语言描述意图 → AI 生成原子化脚本 → 确定性状态机严格执行 → 异常时 AI 决策 / 人工兜底

---

## 1. 引言

### 1.1 项目背景
当前市场上的 RPA（机器人流程自动化）工具（如**影刀 RPA**、UiPath、云扩等）普遍存在以下痛点：
- **闭源生态**：核心引擎不开放，难以与 Python 数据科学生态和 AI Agent 深度集成；
- **扩展性差**：自定义逻辑需学习专有 DSL 或拖拽组件，对开发者极不友好；
- **AI 结合弱**：多数产品仅将 AI 作为辅助插件，未将大模型作为原生编排层；
- **反检测能力弱**：浏览器自动化易被反爬策略拦截，且缺乏灵活的降级方案；
- **桌面自动化受限**：对非标准控件、游戏、老旧软件的图像级操作支持有限。

**OmniAuto 的解决思路**：自主从头构建一个以 Python 为核心、以 AI 为编排层、以开源为信仰的通用自动化框架，直接对标影刀 RPA 的自动操作能力，但在**扩展性、AI 相性、Python 生态集成**上全面超越。

### 1.2 文档目的
本文档定义 OmniAuto 框架的全部功能性与非功能性需求，为后续系统架构设计、开发实施与测试验收提供唯一权威依据。

### 1.3 目标用户
- **业务人员**：无编程基础，通过自然语言或低代码界面创建自动化任务；
- **Python 开发者**：基于框架快速扩展新的自动化能力、集成第三方系统（如 Pandas、Scrapy、LangChain）；
- **AI 编程工具**：作为 Codex / Kimi Code 等工具的 Skill 上下文，辅助生成合规代码。

## 2. 功能性需求

### 2.1 通用任务描述能力 (Meta-Programming)

| 编号 | 需求描述 | 验收标准 |
|------|---------|---------|
| **F1.1** | 系统应支持用户用自然语言描述复杂多步骤任务 | 用户输入 "每天早 9 点登录 ERP，导出昨日销售报表，筛选金额大于 1 万的订单，发送到钉钉群"，系统能成功解析并生成对应工作流 |
| **F1.2** | 系统自动将任务拆解为原子操作序列 | 拆解结果至少包含：导航、登录、点击、数据提取、筛选、发送通知等原子步骤 |
| **F1.3** | 支持任务模板的复用与参数化 | 支持变量注入（如日期、用户名）、循环（遍历列表）、条件分支（if/else） |
| **F1.4** | 提供类影刀的低代码/无代码任务编排界面（第一阶段以 CLI 为主，后续扩展 Web IDE） | 用户可通过 `omni generate` + `omni debug` 快速创建、调试、运行工作流 |

### 2.2 多模态自动化引擎

#### 2.2.1 浏览器自动化

| 编号 | 需求描述 | 验收标准 |
|------|---------|---------|
| **F2.1.1** | 支持基于 CDP 协议连接真实 Chrome 实例 | 能连接用户本地已登录的 Chrome Profile，绕过基础反爬检测 |
| **F2.1.2** | 支持 Playwright + Stealth 插件方案 | 在常规反爬网站（如淘宝、京东）上能正常执行点击、输入、提取操作 |
| **F2.1.3** | 支持视觉自动化兜底 | 当浏览器级 API 被检测时，能自动切换为屏幕级物理操作（PyAutoGUI / pyauto-desktop） |
| **F2.1.4** | 支持复杂页面结构 | 能处理多标签页、iframe 嵌套、Shadow DOM |

#### 2.2.2 桌面软件自动化

| 编号 | 需求描述 | 验收标准 |
|------|---------|---------|
| **F2.2.1** | 支持图像识别定位元素 | 基于 OpenCV / pyauto-desktop 模板匹配，在目标窗口中定位按钮/输入框，准确率 > 90% |
| **F2.2.2** | 支持系统级鼠标键盘模拟 | 模拟人类贝塞尔曲线鼠标轨迹，支持点击、拖拽、键盘输入 |
| **F2.2.3** | 支持 OCR 识别非标准控件文字 | 能识别传统 Win32 控件、Java Swing、Qt 等框架中的文字内容 |
| **F2.2.4** | 支持窗口管理 | 能按标题/类名查找窗口，执行切换、最小化、最大化、置顶 |
| **F2.2.5** | 支持跨分辨率/跨 DPI 自动缩放 | 脚本在开发机（如 2K/125% 缩放）与运行机（如 1080P/100% 缩放）间无需修改即可运行 |

### 2.3 AI 编排与决策层

| 编号 | 需求描述 | 验收标准 |
|------|---------|---------|
| **F3.1** | 脚本生成器 | 根据自然语言生成 Python 原子脚本，使用 Smolagents 代码优先模式，输出可直接注册到状态机的脚本文件 |
| **F3.2** | 状态机引擎 | 工作流执行必须基于确定性状态机，**禁止 AI 在运行时直接操作 UI** |
| **F3.3** | 异常决策 | 原子步骤失败时，自动调用 AI 做策略选择：重试 / 换方案 / 转人工 |
| **F3.4** | 人工审核节点 | 高危操作（转账、删除、发送邮件/消息、修改配置）必须强制暂停并等待人工确认 |

### 2.4 反检测与 Stealth 能力

| 编号 | 需求描述 | 验收标准 |
|------|---------|---------|
| **F4.1** | 浏览器指纹伪装 | 覆盖 `navigator.webdriver`、伪造 Plugins、混淆 Canvas 指纹 |
| **F4.2** | 人类行为模拟 | 随机延迟（点击 100-300ms，输入 50-150ms/字符）、非线性鼠标移动、滚动速度变化 |
| **F4.3** | 请求级伪装 | 支持 User-Agent 轮换、代理 IP 池、TLS 指纹模拟 |

## 3. 非功能性需求

| 类别 | 需求 | 指标 |
|------|------|------|
| **稳定性** | 原子操作成功率 | > 95%（含 3 次重试后） |
| **可维护性** | UI 变化适配成本 | 通过 AI 视觉理解自动适配，无需重写 CSS 选择器 |
| **安全性** | 敏感数据隔离 | 凭证加密存储（AES-256），高危操作人工确认 |
| **可扩展性** | 新任务接入成本 | 自然语言描述即可，< 30 分钟生成可用脚本 |
| **性能** | 任务执行延迟 | 浏览器 API 操作 < 3s/步，视觉操作 < 5s/步 |
| **可恢复性** | 断点续传 | 每完成一个原子步骤立即持久化，支持崩溃后从当前步骤恢复 |
| **Python/AI 相性** | 与 Python 生态和 AI Agent 的无缝集成 | 所有原子脚本均为原生 Python，可直接 import 任意 PyPI 库；AI 编排层原生支持 OpenAI / Claude / Ollama |

## 4. 用户角色与使用场景

### 4.1 角色定义

- **任务设计者 (Designer)**：用自然语言描述任务，不负责编码
- **任务审核者 (Guardian)**：审批涉及敏感操作的工作流，处理异常队列
- **系统开发者 (Engineer)**：扩展引擎能力、修复 Bug、优化反检测策略

### 4.2 典型使用场景

1. **电商价格监控**: 每天定时登录淘宝/京东，搜索关键词，提取前 10 个商品的价格和库存，生成 Excel 报表并发送邮件。
2. **ERP 数据同步**: 登录内部 ERP，导出昨日订单，筛选大额订单后自动录入到钉钉审批流。
3. **桌面软件自动化**: 打开财务软件，按日期区间导出报表，通过邮件发送给指定联系人。

## 5. 术语表

| 术语 | 定义 |
|------|------|
| **Atomic Step** | 原子步骤，工作流中不可再拆分的最小执行单元（如点击、输入、提取） |
| **State Machine** | 确定性状态机，负责按预设顺序执行原子步骤，处理状态转换和异常 |
| **Guardian Point** | 人工审核节点，到达此节点时工作流暂停，等待人工确认 |
| **Stealth Engine** | 反检测引擎，负责指纹伪装、行为模拟、请求伪装 |
| **Visual Fallback** | 视觉自动化兜底，当浏览器级自动化失效时切换为图像识别+物理模拟 |
| **Vibe Coding** | 一种通过与 AI 对话即可生成和维护代码的开发方式 |
