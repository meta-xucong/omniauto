# VPS 服务端优化方案 V2

> 2026-05-05 更新：本方案的 patch 下发链路已被“云端正式库唯一来源 + 客户端刷新云端快照”取代。`shared_library` 是唯一正式共享公共知识库，客户端只保留 runtime 只读缓存；历史 patch 概念仅作为兼容命令、审计和旧数据迁移参考。

## 1. 背景

客户端已经收敛为“原始内容 -> AI经验 -> 待确认知识 -> 正式知识”的统一晋升路径。服务端需要承接更上层的共享公共知识治理：

1. 各客户本地系统把具备普适性的内容上传为共享公共知识候选。
2. admin 在服务端审核候选。
3. 审核通过后进入正式共享公共知识库，并生成共享知识补丁。
4. admin 再手动把补丁或版本更新推送给客户电脑。

当前服务端已有基础能力，但页面和链路表达还不够清晰，容易让 admin 误以为“采纳候选”就是客户电脑已更新。

## 2. 现状诊断

### 2.1 已经合理的部分

- VPS 控制台与客户 Local 客户端分离，符合“服务端只做控制平面”的定位。
- admin 是隐藏平台管理员，customer/guest 不会看到 admin。
- 共享公共知识、客户正式知识、商品专属知识、RAG 数据已经分层存放。
- local 节点通过主动轮询命令队列执行备份、还原、检查更新等任务。
- 共享知识候选已经能由 local 上传到 VPS，并由 admin 审核。

### 2.2 需要优化的部分

- 共享公共知识页面把“新增正式库、候选库、正式库”放在一起，但没有清楚呈现“补丁发布”和“推送到客户电脑”。
- admin 采纳候选后会自动生成 patch，但页面没有展示 patch 列表，也没有一键下发 patch 的入口。
- 服务端接受 proposal 时没有先用共享补丁安全规则校验 operations，存在坏路径或坏结构进入发布记录的风险。
- patch 没有按环境变量生成签名；如果 local 端开启签名校验，会出现 VPS 发布了补丁但 local 无法应用的问题。
- 版本更新只支持登记 release，不支持从服务端向客户电脑下发“检查更新/拉取版本”的命令。
- 共享候选详情不够可读，admin 难以看清“来自哪个客户、建议写入哪里、正文是什么、会生成什么补丁”。

## 3. 目标架构

服务端页面按四个动作理解：

1. **候选审核**：从客户上传来的普适知识里挑选可采纳内容。
2. **正式共享库**：服务端已经确认的公共知识。
3. **补丁发布**：每次采纳候选或手动新增共享知识后，都形成可追踪的 patch。
4. **推送到客户电脑**：admin 明确点击后，服务端才向 local 节点创建 `pull_shared_patch` 或 `check_update` 命令。

## 4. 具体优化

### 4.1 共享公共知识页面

- 增加流程提示：候选待审 -> 正式共享库 -> 共享补丁 -> 推送客户电脑。
- 候选卡片增加“查看详情”。
- 详情里展示：
  - 来源客户
  - 提交时间
  - 建议写入路径
  - 建议内容摘要
  - operations 技术清单折叠展示
- patch 列表常驻展示：
  - patch 版本
  - 来源 proposal
  - 包含条目
  - 是否已签名
  - 推送按钮

### 4.2 服务端 patch 安全

- 接受 proposal 前先调用共享补丁安全校验。
- 只允许 `upsert_json`。
- 只允许写入共享知识根目录下的 JSON 文件。
- proposal 至少要产生一个正式共享知识条目，否则拒绝采纳。
- 若配置 `WECHAT_SHARED_PATCH_SECRET`，VPS 发布 patch 时自动签名。

### 4.3 推送共享知识补丁

- 新增服务端接口：
  - `POST /v1/admin/shared/patches/{patch_id}/push`
- 行为：
  - 默认推送给所有已注册客户电脑。
  - 可按 `tenant_id` 或 `node_id` 过滤。
  - 为每个目标节点创建 `pull_shared_patch` 命令。
  - 命令 payload 带完整 patch，local 可直接 preview/apply。

### 4.4 推送版本更新

- release 增加可选字段：
  - `sha256`
  - `signature`
  - `notes`
- 新增服务端接口：
  - `POST /v1/admin/releases/{release_id}/push`
- 行为：
  - 为目标节点创建 `check_update` 或 `push_update` 命令。
  - 当前默认只通知 local 检查更新，不自动安装。

### 4.5 local 命令处理

- `pull_shared_patch` 不再只是 deferred：
  - 如果命令 payload 含完整 patch，则本地先 preview，再 apply。
  - 如果没有完整 patch，则保持 deferred，并报告缺少 patch。
- `check_update` 保持只检查、不安装。
- `push_update` 当前保持 deferred，等待更新包签名/安装器能力完善。

## 5. 不做的事

- 不开放真实 destructive restore。
- 不自动安装更新包。
- 不允许 customer 直接写服务端正式共享库。
- 不把 VPS 控制台变成客户日常知识管理入口。

## 6. 验收标准

- 服务端共享候选采纳前有 patch 安全校验。
- 采纳候选后正式共享库和 patch 都能看到。
- admin 可以把 patch 推送成 local 命令。
- local 收到含 patch 的 `pull_shared_patch` 命令后可预览并应用。
- release 可登记校验信息，并能创建检查更新命令。
- 现有 VPS 控制平面测试、共享知识测试、多租户同步测试全部通过。
- 服务端页面桌面和移动端无横向溢出，按钮和卡片不过大、不粘连。
