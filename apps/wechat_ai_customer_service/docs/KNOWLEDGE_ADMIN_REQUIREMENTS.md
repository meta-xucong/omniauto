# 微信 AI 客服知识管理台需求文档

## 1. 背景

当前微信 AI 客服已经具备：

- 基于 `OmniAuto` 的微信窗口操作、监听与回复流程；
- 结构化知识库：商品、FAQ、公司信息、开票、物流、售后、风格话术；
- 原始资料目录：聊天记录、商品资料、政策文件、ERP 导出；
- AI 候选生成目录：`pending / approved / rejected`；
- 离线回归、文件传输助手 live regression、DeepSeek 边界测试。

下一阶段目标是做一个面向业务用户的本地 Web 管理台，后续可以封装成桌面应用客户端。用户不需要直接编辑 JSON 文件，也不需要理解程序目录，即可查看、修改、导入、审核、检测和应用客服知识。

## 2. 目标

知识管理台要解决四类问题：

1. 用户能看懂当前客服系统“知道什么、按什么规则回复、遇到什么情况转人工”。
2. 用户能安全地新增、修改、删除规则和关键知识。
3. 用户能上传原始文件，让 AI 辅助抽取结构化候选知识，但候选必须经过确认后才能进入正式库。
4. 用户能一键检测当前知识结构是否合理，并看到可执行的优化建议。

## 3. 设计原则

- 正式知识库不能被 AI 直接覆盖。AI 只能生成候选，用户确认后晋升。
- 所有修改必须可预览、可校验、可回滚。
- 正式库、草稿、候选、原始文件、运行时日志必须分层存放。
- 管理台只管理微信 AI 客服应用的专用知识，不污染 OmniAuto 底层。
- 检测结果必须能被业务用户理解，不能只输出程序错误。
- 支持自然语言增量训练，但最终落地仍要转成结构化知识。
- 支持低 token 调用：只加载与当前页面、当前检测、当前学习任务有关的知识。

## 4. 用户角色

- 操作员：查看知识、上传文件、发起 AI 学习、查看检测结果。
- 管理员：新增、修改、删除、应用候选、回滚版本、运行全量检测。

第一版可以不做复杂账号系统，但需要预留角色字段。默认运行在本机 `localhost`，只给本机用户使用。

## 5. 功能范围

### 5.1 首页总览

首页展示：

- 当前知识库版本号、最近修改时间、最近检测结果；
- 商品数量、FAQ 数量、政策模块数量、风格话术数量；
- 待审核候选数量；
- 最近上传文件数量；
- 最近一次自动测试结果；
- 人工接管样本数量；
- 当前客服运行配置摘要，例如目标会话、LLM 是否启用、限流设置。

首页不显示复杂 JSON，重点让用户知道系统当前是否健康。

### 5.2 知识查看

用户可查看当前正式知识，包括：

- 商品信息：名称、别名、品类、价格、起订量、库存、发货、保修、规格、阶梯价、备注。
- FAQ：意图、关键词、优先级、答案、是否需要人工接管。
- 公司信息：公司名、地址、营业时间、客服电话、税号、对公账户等测试字段。
- 开票规则：支持票种、所需字段、开票时间、红冲重开规则。
- 支付规则：支付方式、定金、尾款、账期边界。
- 物流规则：仓库、承运商、发货时效、偏远地区、上楼规则。
- 售后规则：质保、破损、退换货、退款赔偿边界。
- 风格话术：寒暄、报价、折扣、资料缺失、未知问题、转人工等示例。
- 人设与回复边界：客服角色、语气、不可承诺内容、必须转人工内容。

每一页应有搜索、筛选、排序和“查看原始 JSON”入口。

### 5.3 手动编辑

用户可通过表单安全修改知识：

- 新增商品、编辑商品、删除商品；
- 新增 FAQ、编辑 FAQ、删除 FAQ；
- 编辑公司、开票、支付、物流、售后政策；
- 编辑风格话术和人设提示词；
- 编辑关键词、别名、转人工条件；
- 调整优先级和 token 预算。

每次修改默认进入草稿状态。用户点击“应用”前，系统必须：

- 展示差异对比；
- 运行结构校验；
- 检查重复 ID、重复别名、关键词冲突；
- 生成版本快照；
- 写入审计日志。

### 5.4 原始文件上传

用户可上传：

- 聊天记录：`.txt`、`.md`、`.json`、`.csv`；
- 商品表：`.csv`、`.xlsx`、`.json`；
- 规则政策：`.txt`、`.md`、`.docx` 后续支持；
- ERP 导出：`.csv`、`.xlsx`、`.json`。

第一版建议先支持现有脚本已覆盖的 `.txt / .md / .json / .csv`，再扩展 `.xlsx / .docx / PDF`。

上传时需要用户选择资料类型：

- 聊天记录；
- 商品资料；
- 政策规则；
- ERP 数据；
- 未分类。

系统保存到：

- `data/raw_inbox/chats/`
- `data/raw_inbox/products/`
- `data/raw_inbox/policies/`
- `data/raw_inbox/erp_exports/`

上传记录要写入运行时索引，方便用户看到“这个文件是否已经学习过”。

### 5.5 AI 学习

用户点击“AI 学习”后，系统执行：

1. 读取用户选中的原始文件；
2. 根据文件类型和内容标签，选择最小必要上下文；
3. 调用本地规则抽取或 DeepSeek；
4. 生成结构化候选；
5. 候选进入 `data/review_candidates/pending/`；
6. 前端展示候选摘要、证据来源、建议写入位置、风险提示；
7. 用户可批准、编辑后批准、拒绝。

AI 学习结果不得直接写入正式库。

候选类型包括：

- 新增商品；
- 更新商品字段；
- 新增或更新 FAQ；
- 新增风格话术；
- 新增关键词或别名；
- 新增转人工规则；
- 需要人工判断。

每个候选必须带：

- 来源文件；
- 原文摘录；
- 内容 hash；
- 建议目标文件；
- 变更类型；
- 建议字段；
- 置信度；
- 风险等级；
- 建议测试。

### 5.6 候选审核与晋升

用户可在候选页执行：

- 查看候选；
- 查看原始证据；
- 编辑候选字段；
- 批准并应用；
- 拒绝；
- 标记为需要人工补充；
- 合并多个候选。

应用候选前必须：

- 生成正式库快照；
- 执行 schema 校验；
- 执行冲突检测；
- 执行相关回归测试；
- 展示会影响哪些问答。

应用成功后：

- 候选移动到 `approved`；
- 正式库更新；
- `manifest.json` 必要时更新；
- 写入审计日志；
- 首页健康状态刷新。

拒绝后：

- 候选移动到 `rejected`；
- 记录拒绝原因；
- 不影响正式库。

### 5.7 一键检测

一键检测分为快速检测和全量检测。

快速检测：

- JSON 格式；
- schema 校验；
- 必填字段；
- 重复 ID；
- 重复别名；
- 空答案；
- FAQ 优先级冲突；
- 明显危险规则；
- token 预算估算。

全量检测：

- 快速检测全部内容；
- 离线回归；
- 工作流守护逻辑；
- DeepSeek 小探针；
- 可选文件传输助手 live regression；
- 检查待审核候选；
- 检查人工接管样本；
- 检查知识覆盖率。

检测报告应面向用户展示：

- 总体状态：可用 / 有警告 / 不建议启用；
- 问题列表；
- 影响范围；
- 推荐修复；
- 可一键应用的修复候选；
- 需要人工判断的问题。

### 5.8 优化建议应用

检测发现的问题可以分成三类：

- 自动可修：格式、排序、重复空格、明显字段缺失提示。
- AI 建议修：补充别名、补充 FAQ、补充风格话术。
- 只能人工判断：价格、授权折扣、账期、合同、投诉赔偿、违法违规发票。

用户点击“应用建议”时，仍然进入候选审核流程，不应绕过候选阶段直接写正式库。

### 5.9 版本、备份与回滚

每次应用正式知识变更前，系统要创建快照：

- `data/versions/<timestamp>/structured/`
- 快照说明；
- 操作人；
- 变更摘要；
- 关联候选 ID；
- 检测结果。

用户可以：

- 查看版本历史；
- 比较两个版本差异；
- 回滚到某个版本；
- 回滚后自动运行快速检测。

### 5.10 审计日志

需要记录：

- 文件上传；
- AI 学习任务；
- 候选生成；
- 候选批准、拒绝、编辑；
- 正式库修改；
- 检测任务；
- 版本回滚；
- 测试执行。

审计日志存放在 `runtime/apps/wechat_ai_customer_service/admin/audit.jsonl`。

### 5.11 运行状态与安全提醒

管理台应提示：

- 当前是否正在监听微信；
- 是否有 live test 正在运行；
- 是否有锁文件；
- 当前是否启用真实联系人发送；
- 是否存在未处理人工接管样本；
- 是否存在失败测试。

当用户准备应用知识变更时，如果客服监听进程正在运行，第一版建议提示“建议暂停监听后应用”，避免运行中读取到半更新状态。

## 6. 页面建议

第一版页面：

- 总览；
- 商品知识；
- FAQ 与政策；
- 风格与人设；
- 上传资料；
- AI 学习任务；
- 候选审核；
- 一键检测；
- 版本历史；
- 系统状态。

每个页面避免直接暴露复杂目录，保留“高级：查看 JSON / 查看文件路径”。

## 7. 不够严谨处与补充建议

原始设想是正确的，但需要补充以下边界：

- 必须有草稿和版本管理，否则业务用户误删价格或规则会直接影响客服。
- AI 学习不能一键写正式库，否则错误资料、过期报价、客户随口说的话都可能被当成事实。
- 删除规则前要展示影响范围，例如哪些测试、哪些意图会受到影响。
- 修改商品价格、折扣、账期、赔偿、发票类规则要更谨慎，建议强制二次确认。
- 检测结果要分级，不是所有警告都阻止应用。
- 需要保留“仅检测不应用”的工作流，避免用户误操作。
- 需要记录来源证据，否则后续不知道某条知识从哪里来的。
- 需要处理并发，避免客服监听和管理台同时写正式库。
- 需要预留权限，即使第一版只是本机使用。
- 需要预留导出能力，方便备份或迁移到 ERP。

## 8. 第一版验收标准

第一版完成后应满足：

- 能在浏览器打开本地管理台；
- 能查看正式商品、FAQ、政策、风格话术；
- 能新增和编辑至少商品、FAQ、风格话术；
- 应用变更前能看到差异、校验结果和快照；
- 能上传原始 `.txt / .md / .json / .csv` 文件；
- 能生成待审核候选；
- 能批准候选并写入正式库；
- 能拒绝候选；
- 能运行快速检测；
- 能运行离线回归和工作流守护检查；
- 能查看检测报告；
- 能回滚最近一次正式知识变更；
- 所有管理台操作不影响 OmniAuto 底层目录结构。
