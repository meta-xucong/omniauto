# 智能知识摄入优化方案

## 目标

让 Web 管理台的两条入口具备一致的知识判断能力：

- 知识生成器：用户用自然语言描述，系统判断应进入哪个知识库、还缺哪些关键信息、是否存在风险。
- 资料上传：Excel/CSV/TXT/JSON 等文件导入后，系统拆分出多条候选，并逐条判断是否可入库、是否需补充、是否有超出当前结构的细节需要保留。

## 当前问题

1. 知识生成器已有 DeepSeek + 本地规则合并，但缺少统一的“摄入质检结果”结构。
2. 文件导入候选主要由确定性解析产生，能拆出商品/话术/政策，但没有系统性标注“缺失字段”和“补充问题”。
3. 文件里有额外列或很详细的说明时，部分信息可能不会进入正式知识库的可见字段。
4. 候选审核界面没有明确区分“可应用候选”和“待补充候选”。

## 总体方案

新增一个统一的 `knowledge_intake` 质检层，供知识生成器和文件导入共同调用。

### 1. 分类判断

系统根据以下来源综合判断门类：

- 用户选择的资料类型或知识生成器指定门类。
- 内容关键词、表头、字段结构。
- DeepSeek 输出的 `category_id`。
- 本地规则兜底分类。

当 DeepSeek 可用时，它参与自然语言生成器的分类和结构化；当 DeepSeek 不可用时，本地规则保证离线可测、可用。

资料上传入口也支持可选的 DeepSeek 整文抽取：先让模型按文件内容拆成多条候选，再逐条进入本地质检层。即使模型抽取失败，系统也会退回确定性解析，避免上传学习入口整体不可用。

DeepSeek 抽取结果必须经过“来源锚定”检查：商品名、政策答案、客服话术或 ERP 关键字段需要能在原文里找到依据。若模型生成了原文不存在的商品、价格、库存或承诺，该候选会被丢弃，系统回退到本地解析。

### 2. 缺失字段检测

每个门类有两类关键字段：

- schema 必填字段：例如政策的 `title`、`policy_type`、`answer`。
- 业务必需字段：例如商品至少需要 `name`、`price`、`unit`，ERP 记录至少需要来源系统、记录类型、外部编号。

缺失时不丢弃，而是生成 `needs_more_info` 状态：

- 知识生成器：继续保持会话，向用户追问缺失字段。
- 文件导入：候选进入待审核队列，但标记为“待补充”，禁止直接应用入库。

当一个 TXT/MD 文件里同时包含多个带标签的段落，例如“商品资料”“政策规则”“客服话术”，本地解析会逐段拆分成多个候选；完整候选可审核入库，缺字段候选暂存在待补充队列。

### 3. 额外信息保留

每个默认知识库增加 `additional_details` 字段，用于保存当前结构无法容纳的内容：

- Excel/CSV 里的未知列。
- 用户自然语言里暂时没有对应字段的补充说明。
- 原始资料摘录。

`additional_details` 会被管理台以业务表格展示，并进入知识检索与客服证据包，避免细节沉没。

### 4. 风险判断

质检层统一识别高风险信息：

- 账期、月结、赔偿、免单、虚开发票、最低价、保价、保证效果等。
- 商品资料中的风险说明默认作为提醒，不一定整条禁用自动回复。
- 明确“不能自动回复/必须转人工/整单转人工”时才强制转人工。

对于客服对话中的超界诉求，DeepSeek 返回格式异常时也不能失败放行。系统会使用启发式边界判断生成合规的“请示上级/转人工”候选，保证破例优惠、账期、赔偿、虚开发票等场景始终走人工决策。

### 5. 候选审核交互

候选列表需要展示：

- 可应用
- 待补充
- 高风险

候选详情需要展示：

- 推荐入库门类。
- 缺失字段和系统建议追问语。
- 保留下来的额外信息。
- 风险提示。

待补充候选不能直接应用，用户应先补齐信息后用知识生成器或手动新增知识生成正式条目。

## 验收标准

- 自然语言输入缺少价格时，生成器进入收集中状态并提示补价格。
- 文件上传缺少商品价格时，生成候选但标记待补充，应用被阻止。
- 文件上传到错误门类但内容明显是政策时，候选应进入政策库。
- 一个混合文本文件可拆成多个商品/政策/话术候选。
- LLM 生成的候选若无法在原文中锚定，应被拒绝并回退本地解析。
- Excel 中额外列能进入 `additional_details`，应用后在产品库详情可见。
- 所有默认门类都有 `additional_details` 字段，并参与检索/回复证据。
- DeepSeek 对超界诉求输出异常时，系统仍能兜底生成转人工建议。
- 管理台全量回归通过。
