# RAG 经验知识库落地方案

## 目标

本方案用于把“AI 参考资料”和“对话中学到的经验”整理成一个可管理、可去重、可升级、可废弃的 RAG 经验知识库，同时继续保持正式知识库的最高优先级。

最终用户应能在管理台中做到：

1. 看到已经导入 RAG 的资料源、切片概览、索引状态。
2. 看到所有 RAG 经验，而不是只看到统计数字。
3. 看到每条 RAG 经验和正式知识库的关系：已有正式知识覆盖、可能冲突、值得升级、已升级、已废弃。
4. 一键废弃不合适的 RAG 经验。
5. 一键把有价值的 RAG 经验转为“待确认知识”，由用户审核后再进入正式知识库。
6. 保证 RAG 经验不会绕过正式审核自动写入正式知识库。

## 核心原则

### 正式知识优先

客服运行时的回答顺序保持不变：

1. 安全规则、人工接管规则、明确禁止项。
2. 正式结构化知识库：商品、政策、话术、商品专属 FAQ/规则/解释。
3. RAG 参考资料和 RAG 经验：只提供辅助证据、语气润色和弱关联参考。
4. 大模型自由推理：只在允许范围内补充表达，不能凭空承诺价格、赔偿、账期、合同、售后责任。

### RAG 经验不是正式规则

RAG 经验默认可以进入检索增强层，帮助客服更像真人地表达，但不能直接授权业务承诺。需要固定下来的经验，必须通过“升级为待确认知识 -> 用户审核 -> 应用入库”的流程进入正式知识库。

### 用户无脑管理

用户不需要理解 JSON、向量、chunk、embedding 等技术概念。前端只展示：

- 资料源有多少；
- 资料切片是否正常；
- 哪些对话经验可保留、可废弃、可升级；
- 哪些经验已经被正式知识覆盖；
- 哪些经验可能和正式知识冲突。

## 数据关系

### 正式知识库

正式知识库仍按三层结构存放：

1. 平台通用知识。
2. 当前客户通用知识。
3. 当前客户商品专属知识。

正式知识库是可审核、可回滚、可检测、可编译的权威层。

### RAG 资料源

RAG 资料源来自用户上传、聊天记录、ERP 导出、产品说明、政策文档等。它们被拆成切片并建立索引。资料源可以用于检索和候选审核证据，但不等于正式知识。

### RAG 经验

RAG 经验来自客服真实对话中使用 RAG 资料成功回答后的概括。它有独立生命周期：

- `active`：默认采纳，参与 RAG 检索。
- `discarded`：用户废弃，不再参与检索。
- `promoted`：已转为待确认知识，等待用户在候选审核中处理。

管理台会在展示时动态计算它与正式知识库的关系：

- `novel`：尚未被正式知识覆盖，可作为新经验保留。
- `covered_by_formal`：已有正式知识覆盖，应降低优先级或仅保留为语气参考。
- `supports_formal`：和正式知识方向一致，可作为辅助表达。
- `conflicts_formal`：疑似和正式知识冲突，需要人工检查。
- `promotion_candidate`：使用次数较多、无高风险词、未被正式知识覆盖，建议升级为正式知识候选。
- `promoted`：已经升级为候选。
- `discarded`：已经废弃。

## RAG 经验与正式知识的协作

### 检索时

正式知识先决策，RAG 只补充：

- 解释型内容；
- 闲聊时的语气参考；
- 非关键细节的补充材料；
- 用户提问不完全匹配标准库时的弱召回。

当 RAG 经验与正式知识重复时，正式知识优先。RAG 经验可以继续作为“说法参考”，但不能覆盖正式答案。

### 管理时

管理台需要显示所有 RAG 经验，允许用户：

- 保留默认经验；
- 废弃错误经验；
- 升级为正式知识候选；
- 通过候选审核补充字段、切换门类、应用或拒绝。

### 检测时

一键检测后续可扩展为检查：

- RAG 经验和正式知识重复；
- RAG 经验和正式知识冲突；
- 高风险 RAG 经验是否仍处于 active；
- 多条 RAG 经验是否相互矛盾；
- 值得升级的经验是否长期未处理。

本轮先落地管理台可视化、关系标注和升级候选入口。

## API 设计

### `GET /api/rag/sources`

返回 RAG 资料源与切片预览，供“已导入资料”页面展示。

核心字段：

- `sources`：资料源列表。
- `chunks`：切片预览。
- `chunk_counts`：每个资料源切片数量。

### `GET /api/rag/experiences?status=all`

返回全部 RAG 经验，并为每条经验动态附加：

- `formal_relation`：和正式知识的关系。
- `formal_match`：命中的正式知识条目摘要。
- `recommended_action`：建议保留、废弃、升级或检查冲突。

### `POST /api/rag/experiences/{experience_id}/discard`

把经验标记为废弃，并重建 RAG 索引。

### `POST /api/rag/experiences/{experience_id}/promote`

把经验生成一条 pending review candidate。用户仍需要到“待确认知识”中审核后才能正式入库。

## 前端设计

### 已导入资料

展示：

- 资料源数量、切片数量、索引状态、经验数量。
- 资料源列表：类型、门类、商品 ID、切片数、来源路径。
- 切片预览：只展示前若干条，避免技术细节过载。
- 检索测试入口：用户可以输入问题，看系统命中哪些资料。

### 对话中学到的经验

展示：

- 全部经验计数；
- 默认采纳数；
- 已废弃数；
- 已升级数；
- 建议升级数；
- 经验列表。

每条经验展示：

- 经验概括；
- 客户问法；
- 回复要点；
- 命中资料；
- 与正式知识关系；
- 建议操作；
- 废弃按钮；
- 升级为正式知识按钮。

## 验收标准

1. RAG 资料源可以在前端看到具体列表，不再只有数字。
2. RAG 经验可以看到全部状态，不再只能看到 active。
3. active 经验仍参与 RAG 检索，discarded/promoted 经验不参与。
4. 经验可以一键废弃。
5. 经验可以一键升级为 pending candidate，但不会直接入正式库。
6. 经验列表能标注与正式知识的覆盖、冲突、升级建议关系。
7. PostgreSQL 和 JSON fallback 模式的 admin 回归测试通过。
8. 现有客服运行逻辑、安全边界和候选审核流程不被破坏。
