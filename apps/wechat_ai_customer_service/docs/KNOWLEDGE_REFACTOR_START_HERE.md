# 分类知识库重构开工索引

本文档是下一轮代码改造的入口。开工前先读本文件，再按阅读顺序进入细分指导文档。

## 1. 本轮目标

把微信 AI 客服系统从“旧大 JSON 知识文件 + 开发者后台”升级为：

- 后端正式知识源按门类物理隔离；
- 微信客服运行时直接读取分类知识库；
- 当前已有知识迁移到正确门类；
- AI 学习、候选审核、检测、备份恢复都围绕分类知识库工作；
- Web 管理台面向普通业务用户，以表单、卡片、摘要为主，不默认展示 JSON；
- 支持客户自定义知识门类。

## 2. 推荐阅读顺序

1. `KNOWLEDGE_BASE_RUNTIME_REFACTOR_PLAN.md`  
   总体架构、阶段顺序和不可破坏约束。

2. `KNOWLEDGE_REFACTOR_IMPLEMENTATION_CHECKLIST.md`  
   逐章代码落地清单，每章改哪些文件、验收什么。

3. `KNOWLEDGE_BASE_SCHEMA_SPEC.md`  
   分类知识库目录、注册表、schema、item、custom 门类格式。

4. `KNOWLEDGE_RUNTIME_INTERFACE_SPEC.md`  
   微信客服运行时如何读取分类知识、构造 evidence pack、生成 reply plan。

5. `KNOWLEDGE_MIGRATION_TEST_PLAYBOOK.md`  
   旧知识迁移、 dry-run、备份、回滚、测试命令。

6. `ADMIN_FORM_UX_API_SPEC.md`  
   Web 管理台表单化改造、API 合约、候选审核、一键检测、备份还原。

## 3. 开工纪律

- 先改后端结构，再改微信客服运行时，最后改前端体验。
- 每完成一章，必须跑本章测试，再跑相关回归。
- 不允许把前端做成“看起来分类”，但后端仍把知识混在一起。
- 不允许把旧 `product_knowledge.example.json` 继续作为唯一正式知识源。
- 不允许 AI 学习结果绕过候选审核直接入正式库。
- 不允许正式入库、还原、回滚前不备份。
- 不允许普通用户默认看到大段 JSON。
- 不允许高风险业务问题被“一键修复”自动改掉。

## 4. 推荐执行方式

每章遵循：

1. 阅读本章指导文档。
2. 列出本章文件范围。
3. 实现最小可运行变更。
4. 跑窄测试。
5. 修复失败。
6. 跑相关回归。
7. 更新 long-running task 状态和测试日志。
8. 进入下一章。

## 5. 交付判断

只有同时满足以下条件，才算完成本轮重构：

- `data/knowledge_bases` 是正式知识源；
- 默认门类和自定义门类都可创建、校验、读写；
- 微信客服流程直接基于分类 evidence pack 决策；
- 旧 structured/compiled 只作为兼容导出或缓存；
- 当前知识已迁移完成且可对账；
- Web 管理台不依赖 JSON 编辑完成日常操作；
- 自动化测试和页面 E2E 均通过；
- 没有测试上传、测试候选、测试版本快照残留。
