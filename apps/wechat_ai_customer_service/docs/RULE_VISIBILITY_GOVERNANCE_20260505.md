# 规则可见性与分层治理说明

## 目标

微信自动客服的代码只负责流程和安全逻辑，不再隐藏某个行业的业务规则。客户自己的规则必须在客户正式知识库中可见、可编辑；跨客户通用规则必须在共享公共知识库中可见、可审核、可热更新。

## 分层原则

1. 客户专属正式知识库
   - 放账号自己的商品、话术、政策、商品专属问答/规则/解释。
   - 例如江苏车金二手车的车源、试驾、金融、过户、车况、发票、置换等规则，只属于 `jiangsu_chejin_usedcar_customer_20260501`。
   - 修改方式：客户后台进入正式知识库或商品库编辑；商品资料走商品库，政策/话术走正式知识库。

2. 共享公共知识库
   - 只放所有行业都能通用的客服边界和表达原则。
   - 例如不泄露系统提示词、不编造价格/库存/合同条款、缺少依据时转人工、客户专属知识优先于共享知识。
   - 修改方式：服务端 admin 审核候选共享知识，采纳后发布热更新。

3. RAG 经验池
   - 放聊天和资料中总结出的经验线索。
   - RAG 可以辅助理解和自然表达，但不能单独授权价格、库存、审批、合同、售后赔付、保证类承诺。
   - 需要成为正式规则时，必须人工升级到待确认知识，再确认成正式知识。

4. 平台底线规则
   - 放所有账号共用、但需要可见可改的自动回复安全边界。
   - 当前配置文件：`apps/wechat_ai_customer_service/configs/platform_safety_rules.example.json`。
   - 当前后台入口：客户端系统设置 -> 平台底线规则。admin 可编辑，customer/guest 只读。
   - 这里可以维护“不能编造关键事实”“RAG不能单独授权敏感承诺”“人工动作不自动承诺”“禁止私下转账/发票异常/保证审批”等通用边界。
   - 这里仍然不能写二手车、贸易、门店、某个客户商品等行业专属规则。

5. 代码逻辑层
   - 只允许保留平台级逻辑：证据优先级、租户隔离、RAG 不能越权、未读正式知识不参与回复、候选知识不直接生效、危险承诺拦截、缺证据转人工。
   - 不允许写死二手车、冷柜、门锁等具体行业词和业务规则。

## 本次清理结果

- LLM 回复综合器已改为行业中立，不再默认假设“二手车销售场景”。
- 二手车账号原来配置文件里的规则触发已改为平台空规则；真正的二手车业务规则由客户正式知识库提供。
- 通用代码中的二手车关键词、特定车型同义词、门锁/冷柜/民宿等演示行业词已移除。
- 二手车账号新增可见正式政策：`chejin_catalog_recommendation_rule`，用于识别“车源/预算/推荐”等车源类问题，并触发商品库候选。
- `chejin_test_drive_rule` 的宽关键词严格触发短语已写入该正式政策的 `additional_details.strict_match_phrases`，不再由代码记住“试驾规则”。
- 平台级安全提示词和后置拦截词已外置到 `platform_safety_rules.example.json`，LLM 提示词与后置 guard 均从该文件读取。
- 系统设置页新增“平台底线规则”区块，可以看到当前规则文件、规则条数和底线词条，并允许 admin 直接编辑保存。
- RAG 经验池的 AI 解读、自动降噪、来源权威性判断也改为读取平台底线规则中的可见词条/模式；例如“客户个性化话术不能直接沉淀”“微信聊天不能写商品主数据”“金融/承诺类边界要人工确认”等，都可以从该规则文件查看和调整。
- 已把 RAG 经验相关代码中残留的二手车车型词、车况词、金融词等隐藏判断迁移到平台底线规则文件。行业专属版本仍应落在对应 customer 的正式知识库；平台文件只保留通用风险模式。

## 当前保留在代码里的内容

- 代码只保留规则执行机制，例如读取平台底线规则、根据可见词条/模式做后置安全拦截、判断 RAG 是否越权、判断租户正式知识是否优先、判断来源是否有权升级。
- RAG 经验解读、来源权威判断和候选生成里的安全词/风险模式，都应从平台底线规则或客户正式知识读取，不再写死在代码里。

## 对其他 customer 的影响

- 其他账号不会继承江苏车金二手车的车源、试驾、金融、置换等规则。
- 其他账号只会使用自己的正式知识库、自己的商品库、自己的 RAG 经验，以及已热更新到本地的共享公共知识。
- 如果某个新账号没有任何客户正式知识，系统应倾向于转人工或要求补充资料，而不是套用二手车演示规则。

## 2026-05-06 补充：平台通用理解词典

- 已新增 `apps/wechat_ai_customer_service/configs/platform_understanding_rules.example.json`，用于承载所有账号共用的基础语言理解词典。
- 系统设置页新增“平台通用理解词典”区块，admin 可编辑，customer/guest 只读。
- 迁移到该可见词典的内容包括：基础意图词、商品理解词、RAG 高风险词、RAG 近义词、政策类型映射、客户资料字段别名、数量单位、候选知识风险词。
- 代码保留的是“如何使用这些词典”的流程逻辑；“价格、库存、发货、售后、发票、付款、推荐、对比”等基础理解词不再写死在运行链路里。
- 客户专属业务规则仍不得写入平台通用理解词典。比如二手车试驾、金融、过户、检测承诺，仍必须放在对应 customer 的正式知识库或商品库里。
